  方案分析

  trim_messages 核心参数

  ┌─────────────────────┬────────────────────────────────────────────────────┐
  │        参数         │                        作用                        │
  ├─────────────────────┼────────────────────────────────────────────────────┤
  │ strategy="last"     │ 保留最近的 N 轮，这是关键                          │
  ├─────────────────────┼────────────────────────────────────────────────────┤
  │ max_tokens          │ 控制总 token 上限                                  │
  ├─────────────────────┼────────────────────────────────────────────────────┤
  │ token_counter       │ 计 token 的方式："approximate" 快速估算 /          │
  │                     │ ChatOpenAI(model=...) 精确计 / len 算消息条数      │
  ├─────────────────────┼────────────────────────────────────────────────────┤
  │ start_on="human"    │ 保证裁剪后以 HumanMessage 开头（或 SystemMessage + │
  │                     │  HumanMessage），不会截断一轮对话                  │
  ├─────────────────────┼────────────────────────────────────────────────────┤
  │ include_system=True │ 保留 system prompt                                 │
  ├─────────────────────┼────────────────────────────────────────────────────┤
  │ allow_partial=False │ 不截断单条消息（语义完整性）                       │
  └─────────────────────┴────────────────────────────────────────────────────┘

  你的需求：保留最后 N 轮 + 控制 max_token

  trim_messages 本身没有 "N 轮" 参数，它是基于 token
  裁剪的。但可以组合参数完美满足你的需求：

  核心思想：strategy="last" + start_on="human" + include_system=True

  - strategy="last" → 从末尾开始保留消息
  - start_on="human" → 找到最近的 HumanMessage
  作为起点（确保消息以完整的轮次开始）
  - include_system=True → system prompt 不会丢失
  - max_tokens → 最终 token 上限

  这样就会在 token
  预算内，保留尽可能多的完整对话轮次。如果预算宽松，所有轮次都在 →
  全保留；如果预算紧 → 自动丢弃最老的轮次。

  ★ Insight ─────────────────────────────────────
  • start_on="human" 是保证轮次完整性的关键——它会从末尾向上回溯到第一个
  HumanMessage，确保裁剪后的对话不是一个孤立的 AIMessage 开头。
  • include_system=True 在系统 prompt 位于 index 0 时总是保留它，不计入
  max_tokens 的裁剪范围，这是源码实现中特殊处理的逻辑。
  • token_counter 传 len 可按消息条数裁剪（而非 token），适合明确说"保留最近 6
  条消息"的场景。
  ─────────────────────────────────────────────────

● 找到全部源码了。答案很直接：

  token_counter="approximate" 就是 count_tokens_approximately

  链路只有一层：

  "approximate"  →  _TOKEN_COUNTER_SHORTCUTS["approximate"]  →
  count_tokens_approximately(messages)

  计算公式

  每条消息的 token = 4 个东西加起来：

  # 源码第 2264-2405 行，核心公式
  token_count += math.ceil(message_chars / 4.0)     # ① chars / 4
  token_count += 3.0                                 # ② 每条消息固定 +3
  # (如果是 AIMessage 有 tool_calls → 把 repr(tool_calls) 也算进 chars)
  # (如果是 ToolMessage → 把 tool_call_id 也算进 chars)
  # (角色名 "user"/"assistant" 也算进 chars)

  具体来说：

  ┌──────────────────────────────┬──────────────────────────────────────────┐
  │             成分             │                    值                    │
  ├──────────────────────────────┼──────────────────────────────────────────┤
  │ chars_per_token              │ 4.0（1 token ≈ 4 字符）                  │
  ├──────────────────────────────┼──────────────────────────────────────────┤
  │ extra_tokens_per_message     │ 3.0（每条消息额外加）                    │
  ├──────────────────────────────┼──────────────────────────────────────────┤
  │ 角色名                       │ 算进 chars（如 "user" 5 字符 → 5/4=1.25  │
  │                              │ → ceil=2）                               │
  ├──────────────────────────────┼──────────────────────────────────────────┤
  │ tool_calls（AIMessage        │ repr(tool_calls) 的字符数 / 4            │
  │ 有的话）                     │                                          │
  ├──────────────────────────────┼──────────────────────────────────────────┤
  │ tool_call_id（ToolMessage）  │ 算进 chars                               │
  └──────────────────────────────┴──────────────────────────────────────────┘

  为什么它总是低估中文

  英文: "Hello"         → 5 chars → 5/4 = 1.25 → ceil=2  tokens  ✅ 还好
  中文: "北京今天天气很好！" → 8 chars → 8/4 = 2.0  → 2 tokens  ❌ BPE 实际 7-8

  approximate 认为 4 字符 = 1 token
  BPE 认为 1 个汉字 ≈ 1.5-2 tokens

  误差随着中文比例增加而放大。 这也是为什么你之前看到 approximate 计了 323 但
  BPE 计了 517——差了 60%。

  快查 → approximate × 因子
  当结果在预算 10% 以内 → 回退到精确 BPE 计数
  这样减少了 80-90% 的精确计数调用

  但关键是这个因子不能拍脑袋定，得算出来。

  因子怎么来？

  拿我们场景 3 的数据算一下：

  approximate 计数 = 323
  BPE 精确计数   = 517
  因子 = 517 / 323 ≈ 1.6

  但每个消息的因子不一样：
    SystemMessage:  31/19  ≈ 1.63
    HumanMessage:   20/13  ≈ 1.54
    AIMessage(工具): 0/28  ≈ 0    ← 工具调用 content 空，approximate 有固定 28
    ToolMessage:    36/29  ≈ 1.24
    AIMessage(回答): 77/33 ≈ 2.33  ← 长中文 + emoji，差最大

  问题在于因子不固定——工具调用的 AIMessage（content=""）近似法计了 28 但 BPE 计
  0（因为真的为空），而长回答差到 2.33x。一个全局因子 cover 不了所有情况。

  真正可行的模式

  模式 A：校准 + 固定因子

  跑一次校准任务，算出你项目实际对话的加权平均因子，然后一直用这个值。适合对话模
  式稳定的场景。

  else:

      result = trim_messages(msgs, max_tokens=budget,
  token_counter=bpe_token_counter, ...)

● 好问题。看本质：

  token_counter 的本质

  token_counter 是 trim_messages 内部 binary search 的"度量函数"。

  它只有一个要求：给一个更大的消息子集，返回值必须 ≥ 
  给更小子集的值（单调递增）。

  trim_messages 不在乎这个值是什么——token 数、条数、HumanMessage
  数、甚至字符数，都可以。它只做一件事：

  # binary search 的核心逻辑（简化版）
  token_counter(messages[:mid]) <= max_tokens  # ← 唯一用到 token_counter 的地方

  问："前 mid 条消息的计量值，在预算内吗？"

  - token_counter="approximate" → 问：前 mid 条消息的 token ≤ max_tokens？
  - token_counter=len → 问：前 mid 条消息的条数 ≤ max_tokens？
  - token_counter=_round_counter → 问：前 mid 条消息里的 HumanMessage 数量 ≤
  max_tokens？

  为什么 _round_counter 写成那样

  def _round_counter(msgs: list[BaseMessage]) -> int:
      return sum(1 for m in msgs if isinstance(m, HumanMessage))

  msgs 是"当前正在评估的子集"（不是全部消息）。sum(...) 数这个子集里有多少
  HumanMessage。

  _round_counter 回答的问题是：这个子集包含了多少轮对话？

  AIMessage 和 ToolMessage 不计入，它们是"轮次内部的内容"，不是"轮次"本身。

  验证单调性

  msgs[:3]  →  [System, H1, AI]         →  1 个 Human
  msgs[:5]  →  [System, H1, AI, T, AI]  →  1 个 Human（T、AI 不加也不减）
  msgs[:9]  →  [System, H1, AI, T, AI, H2, AI, T, AI]  →  2 个 Human  ✅

  子集越大，返回值不变或增加——单调性成立。

  ★ Insight ─────────────────────────────────────
  token_counter 的本质不是"计 token"，而是"给 trim_messages 
  一个单调的度量函数，让它的 binary search 能判断哪个子集在预算内"。
  你想用什么单位来约束裁剪，就把那个单位写在 token_counter
  里。Token、条数、轮次，只是不同的度量。
  ─────────────────────────────────────────────────

      703 +`_round_counter` 的 `max_tokens=2` 不等于"保留 2 条消息"，而是
          +"保留 ≤ 2 个 HumanMessage 的完整轮次"。                       
      704 +                                                              
      705 +```text                                                       
      706 +11 条消息，3 个 Human                                         
      707 +                    ↓                                         
      708 +max_tokens=2 → 保留 2 个 Human → 顺带他们对应的 AI/Tool → 共 7
          + 条消息                                                       
      709 +                    ↓                                         
      710 +max_tokens=1 → 保留 1 个 Human → 顺带对应的 AI → 共 3 条消息  
      711 +```                                                           
      712 +                                                              
      713 +这就是自定义 `token_counter` 的本质——你改变了"计量单位"。     
      714 +                                                              
      715 +| `token_counter` | `max_tokens=2` 的含义 | 结果条数 |        
      716 +|---|---|---|                                                 
      717 +| `"approximate"` | 总 token ≤ 2 | 几乎全删光 |               
      718 +| `len` | 保留 ≤ 2 条消息 | 0 条（System 吃 1，start_on 砍残余
          +） |                                                          
      719 +| `_round_counter` | 保留 ≤ 2 个 HumanMessage | 7 条（完整的 2
          + 轮） |                                                       
      720 +                                                              
      721 +你说的这个性质是所有 `trim_messages` 调用共享的，不只是 `_roun
          +d_counter`——**`trim_messages` 通过 `token_counter` 来决定"保留
          +什么"的语义，然后通过 `start_on` 保证保留的东西结构完整。**   
      722 +                                                              
      723 +---                                                           
      724 +                                                              
      725 +### 对话三：`token_counter` 的本质                            
      726 +                                                              
      727 +**问：** 为什么函数内部是这么写的呢？                         
      728 +                                                              
      729 +```python                                                     
      730 +return sum(1 for m in msgs if isinstance(m, HumanMessage))    
      731 +```                                                           
      732 +                                                              
      733 +_round_counter 或者说这个场景下的 token_counter 的本质是什么？
      734 +                                                              
      735 +**答：** 好问题。看本质：                                     
      736 +                                                              
      737 +**`token_counter` 是 `trim_messages` 内部 binary search 的"度 
          +量函数"。**                                                   
      738 +                                                              
      739 +它只有一个要求：**给一个更大的消息子集，返回值必须 ≥ 给更小子 
          +集的值**（单调递增）。                                        
      740 +                                                              
      741 +`trim_messages` 不在乎这个值是什么——token 数、条数、HumanMessa
          +ge 数、甚至字符数，都可以。它只做一件事：                     
      742 +                                                              
      743 +```python                                                     
      744 +# binary search 的核心逻辑（简化版）                          
      745 +token_counter(messages[:mid]) <= max_tokens  # ← 唯一用到 toke
          +n_counter 的地方                                              
      746 +```                                                           
      747 +                                                              
      748 +问：**"前 mid 条消息的计量值，在预算内吗？"**                 
      749 +                                                              
      750 +- `token_counter="approximate"` → 问：前 mid 条消息的 token ≤ 
          +max_tokens？                                                  
      751 +- `token_counter=len` → 问：前 mid 条消息的条数 ≤ max_tokens？
      752 +- `token_counter=_round_counter` → 问：前 mid 条消息里的 **Hum
          +anMessage 数量** ≤ max_tokens？                               
      753 +                                                              
      754 +**为什么 `_round_counter` 写成那样**                          
      755 +                                                              
      756 +`msgs` 是"当前正在评估的子集"（不是全部消息）。`sum(...)` 数这
          +个子集里有多少 HumanMessage。                                 
      757 +                                                              
      758 +**`_round_counter` 回答的问题是：这个子集包含了多少轮对话？** 
      759 +                                                              
      760 +AIMessage 和 ToolMessage 不计入，它们是"轮次内部的内容"，不是"
          +轮次"本身。                                                   
      761 +                                                              
      762 +**验证单调性**                                                
      763 +                                                              
      764 +```python                                                     
      765 +msgs[:3]  →  [System, H1, AI]         →  1 个 Human           
      766 +msgs[:5]  →  [System, H1, AI, T, AI]  →  1 个 Human（T、AI 不 
          +加也不减）                                                    
      767 +msgs[:9]  →  [System, H1, AI, T, AI, H2, AI, T, AI]  →  2 个 H
          +uman  ✅                                                      
      768 +```                                                           
      769 +                                                              
      770 +子集越大，返回值不变或增加——**单调性成立**。                  
      771 +                                                              
      772 +> `token_counter` 的本质不是"计 token"，而是"给 `trim_messages
          +` 一个单调的度量函数，让它的 binary search 能判断哪个子集在预 
          +算内"。你想用什么单位来约束裁剪，就把那个单位写在 `token_count
          +er` 里。Token、条数、轮次，只是不同的度量。
